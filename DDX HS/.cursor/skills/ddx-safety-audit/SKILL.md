# Безопасность анализа (Safety Audit)

## Pre-analysis: Zero Point чек-лист (copy-paste)

```
ЧЕК-ЛИСТ "НУЛЕВАЯ ТОЧКА"

ID Случая: [...]  Дата: [...]

1. Аффективное состояние:
   - Настроение (-5..+5): [...]
   - Сильные эмоции, не связанные с работой (да/нет, кратко): [...]

2. Когнитивное состояние:
   - Концентрация (1..10): [...]
   - Спешка / давление дедлайна (да/нет): [...]

3. Физическое состояние:
   - Усталость (1=бодрый, 10=истощен): [...]
   - Значимый дискомфорт (да/нет): [...]

4. Первичный резонанс с материалом:
   - Первые ассоциации/чувства: [...]
   - Напоминает личный/профессиональный опыт? (да/нет, кратко): [...]

5. Итоговая оценка:
   - Готовность к работе: [Высокая / Средняя / Низкая]
   - Идентифицированные риски: [...]
   - План минимизации: [...]
```

**Действия:** если готовность "Низкая" -- отложить работу или усилить частоту аналитических пауз.

## In-process: Протокол аффективного резонанса (Прил.4, 1.3)

При возникновении **сильной** эмоции (скука, раздражение, симпатия, страх, восхищение) во время работы:

1. **Фиксация** в Блоке 4 аннотации: что чувствую, к какому фрагменту.
2. **Вопрос:** "Что если моя эмоция -- не помеха, а диагностический сигнал? Что если текст активно индуцирует это чувство?"
3. **Гипотеза-кандидат** от рефлексии (Принцип генерации 3, Объект C.2, Интенция 4.1).
4. **Валидация:** гипотеза проходит стандартную проверку (Сценарий 2.1.2). Не выше Риска 3 без строгой проверки.

---

## In-process: Обязательная функциональная гипотеза (Прил.4, 2.1)

**Правило:** для каждой H-Pathology ("феномен = дефицит / дисфункция") **обязательно** создать конкурирующую H-Adaptive ("феномен = адаптивная реакция / осмысленная стратегия / функциональное решение").

**Процедура:**
1. Сформулировать H-Adaptive для той же "мишени".
2. Прогнать пару H-Pathology vs H-Adaptive через Fast Falsification (Skill `ddx-hypothesis-lifecycle`, раздел 4).
3. H-Pathology принимается как центральная **только после** аргументированной фальсификации H-Adaptive.

На системном уровне: если вся модель стала pathology-only -> триггер Mode 2 (Skill `ddx-activate-mode`).

---

## In-process: Контекстуальное заземление (Прил.4, 2.3)

**Правило:** ни одна гипотеза о генерализованной устойчивой черте / расстройстве (атрибут "Генерализованная") не может стать **центральной** без прохождения Сценария C3 "Диспозиция vs ситуация" (из `ddx-contextual-playbooks.mdc`).

**Исключение:** если материал ограничен одним контекстом (только рабочая переписка), в заключении обязательна оговорка: "Вывод сделан на основе [контекст] и требует проверки в других сферах."

---

## In-process: Управление высокорисковыми эвристиками (Прил.4, 2.2)

**Правило:** выводы Axis5 уровня 4-5 (Аналоговый / Трансгрессивный) **не включаются** в основные разделы 2-5 заключения как центральные или подтверждённые.

- **Место:** Раздел 6.1 "Рассмотренные альтернативы" -- как эвристически ценные, но спекулятивные модели.
- **Исключение:** вывод **независимо подтвержден** совокупностью данных Риска 1-3.
- Особенно касается всех H-Adv из Режима 1 (состязательная генерация).

---

## Pre-conclusion: Internal Red Teaming (Прил.4, 3.1)

**Когда:** после построения интегративной модели, **перед** написанием заключения.

### Чек-лист атаки на собственную модель

- [ ] Есть ли данные, **противоречащие** центральной модели, которые я не интегрировал?
- [ ] Не является ли модель **overfitting** -- не объясняю ли я шум как сигнал?
- [ ] Какие **скрытые допущения** лежат в основе модели? Что если они ложны?
- [ ] Все выводы Axis5 4-5 прошли **низкорисковую проверку**?
- [ ] Модель "слишком гладкая" / подозрительно стройная? -> триггер **Mode 3** (Skill `ddx-activate-mode`)
- [ ] Наиболее сильная альтернативная мета-гипотеза: [сформулировать и проверить]

---

## Decision: Триггер внешней супервизии (Прил.4, 3.2)

Рассмотреть привлечение внешнего эксперта если:

- [ ] Случай чувствительный (этические / правовые последствия)?
- [ ] Выводы спорные или контринтуитивные?
- [ ] Ставки высоки (судебная экспертиза, кризисное вмешательство)?
- [ ] Сильный личный резонанс выявлен в Zero Point?

Формат: супервизия (более опытный коллега) или интервизия (peer review). Несогласие = сигнал для ревалидации.

Связанные правила: `ddx-safety.mdc`, `ddx-hypothesis-risk.mdc`, `ddx-contextual-playbooks.mdc`, `ddx-advanced-modes.mdc`.
