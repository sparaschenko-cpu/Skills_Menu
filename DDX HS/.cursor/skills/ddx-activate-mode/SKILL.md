# Продвинутые режимы (Activate Mode)

## Дерево решений: какой режим?

| Ситуация | Режим | Вопрос |
|----------|-------|--------|
| Аналитический тупик, H1 vs H0 неразрешим | **Mode 1** Adversarial | "Что если всё совсем не так?" |
| Модель pathology-only, нет ресурсной перспективы | **Mode 2** Resource | "Что работает хорошо?" |
| Модель "слишком гладкая", высокие ставки | **Mode 3** Meta-critique | "Насколько прочна модель?" |

---

## Протокол активации (4 шага, для любого режима)

1. **Идентификация потребности:** что именно не работает в стандартном цикле? (тупик / однобокость / недостаточная надёжность)
2. **Выбор режима:** Mode 1 / 2 / 3 по дереву решений выше.
3. **Формальная активация** через аннотацию:
   - Интенция: **4.1.2 Методологическая рефлексия**
   - Обоснование: почему режим нужен
   - Цель: что режим должен вернуть
4. **План возврата:** результат режима -> объект для обычной валидации (Сценарий 2.1.2) и интеграции (Интенция 3.2).

## Mode 1: Состязательная генерация (Adversarial Generation)

### Workflow -- 4 шага

**Шаг 1: Активация и подготовка**
- Выбрать мишень: ID центральной H1 из СМГ (статус "Рабочая" или "Подтверждённая").
- Создать аннотацию-декларацию (Интенция 1.2.2, Объект B.1, Риск 4-5).
- Все H-Adv по умолчанию = **Axis5 риск 4-5**.

**Шаг 2: Дивергентная генерация**
- Методичный проход по таксономии эвристик:
  - Категории 1-2 (инверсии + границы/контекст + реципрокный вклад): `ddx-adversarial-heuristics.mdc`
  - Категории 3-4 (трансфер + парадокс): `ddx-adversarial-transfer-paradox.mdc`
- Результат: "длинный список" 5-10 H-Adv.
- **Критика запрещена** на этом этапе -- максимальное разнообразие.

**Шаг 3: Конвергентный отбор**
- Скрининг по: минимальное правдоподобие / объяснительная сила / фальсифицируемость.
- "Короткий список" 2-4 -> выбор 1-2 финалистов по элегантности и контрастности к H1.

**Шаг 4: Валидация и интеграция**
- Запуск Fast Falsification (Skill `ddx-hypothesis-lifecycle`, раздел 4) для пары "H1 vs H-Adv".
- **Формальная сравнительная оценка (обязательно):** построить **Матрицу доказательств** (Прил.5, 1.2) для "H1 vs H-Adv":
  - строки: `H1`, `H-Adv`; столбцы: релевантные `A-###`;
  - ячейки: `+` поддержка, `-` фальсификация, `~` уточнение, `0` нейтрально;
  - цель: прозрачность того, *какие именно* свидетельства тянут модель вверх/вниз (и где пробелы).
  Детали/формат: `ddx-smh-advanced.mdc` (Evidence Matrix). Шаблон: Skill `ddx-hypothesis-lifecycle` → `templates.md`.
- **Решение:** сохранение H1 (прочность подтверждена) / замена на H-Adv / синтез новой H2.
- При большом пуле (>7): запустить Воронку Валидации (Skill `ddx-hypothesis-lifecycle`, раздел 6).

## Mode 2: Ресурсный анализ (Resource-Oriented Analysis)

### Workflow -- 3 шага

**Шаг 1: Применение эвристик**

*SWOT-эвристики:*
- **S** (Сильные стороны): навыки, компетенции, позитивные черты, извлечённые уроки.
- **W** (Слабые стороны): переформулировать как точки роста ("Какая сильная сторона -- тень этой слабости?").
- **O** (Возможности): внешние ресурсы (поддержка, среда, неиспользованные возможности).
- **T** (Угрозы): внешние стрессоры, дестабилизирующие факторы.

*Функциональные эвристики:*
- **Поиск исключений:** "Когда проблемы НЕ было?" -> анализ ситуации-исключения -> вывод-ресурс.
- **Рефрейминг симптома:** "Какую потребность симптом пытается удовлетворить?" -> за симптомом = здоровая потребность.

Каждое наблюдение -> аннотация (Блок 2.3: `Салютогенная -> Ресурсный анализ -> [эвристика]`).

**Шаг 2: Построение Ресурсной карты**

SWOT-матрица из 4 квадрантов. Дополнить результатами рефрейминга как комментарии к W.

**Шаг 3: Интеграция с пато-моделью**

Ключевые вопросы:
- **Компенсация:** как S компенсируют W? Адаптивно или ригидно?
- **Защита:** как S и O защищают от T?
- **Уязвимость:** как T "бьют" по W?
- **Потенциал:** как S + O -> нейтрализация T?

Результат -> Разделы 5.3-5.4 заключения (Skill `ddx-write-conclusion`).

Детали эвристик: `ddx-resource-metacritique.mdc`.

## Mode 3: Мета-критика модели (Model Integrity Audit)

### Workflow -- 3 шага

**Шаг 1: Выбрать финальную интегративную модель как объект аудита.**

Интенция: 4.1.2 Методологическая рефлексия.

**Шаг 2: Применить 4 эвристики последовательно:**

1. **Анализ отказов (FMEA):** "Что если ключевое свидетельство окажется неверным? В каких ситуациях модель не работает?"
2. **Анализ чувствительности:** удалить 2-3 ключевых аннотации мысленно -- модель устоит?
3. **Робастная гипотеза:** оставить только факты Риска 1-2, построить минимальную модель. Это "несущий каркас".
4. **Проверка на избыточность (Оккам):** для каждого элемента -- "Если убрать, что потеряется?" Если ничего -- элемент избыточен.

**Шаг 3: Заполнить Паспорт надежности:**

```
ПАСПОРТ НАДЁЖНОСТИ МОДЕЛИ
1. Ядро (робастная гипотеза): [...]
2. Уязвимости и "режимы отказа": [...]
3. Чувствительность к ключевым свидетельствам: [A-___]
4. Границы применимости: [...]
5. Итоговая оценка: [Высокая / Средняя / Низкая] + обоснование
```

Результат -> Раздел 6.2 заключения (Skill `ddx-write-conclusion`).

Детали эвристик: `ddx-resource-metacritique.mdc`.

## Возврат в стандартный протокол

После завершения любого режима:
- Результат (H-Adv / Ресурсная карта / Паспорт надежности) становится **объектом** для обычной валидации и интеграции.
- Обязательно обновить СМГ-реестр (новые гипотезы с пометкой режима генерации).

Связанные правила: `ddx-advanced-modes.mdc`, `ddx-adversarial-heuristics.mdc`, `ddx-adversarial-transfer-paradox.mdc`, `ddx-resource-metacritique.mdc`, `ddx-validation-funnel.mdc`.
