# Аннотирование и кодирование текста (Annotate & Code)

## Шаблон аналитической аннотации (copy-paste)

```
АНАЛИТИЧЕСКАЯ АННОТАЦИЯ

ID Аннотации: A-___
Источник: [документ / стр. / строки]

Исходный фрагмент текста:
> [прямая цитата]

БЛОК 1: КОНТЕКСТУАЛИЗАЦИЯ (DDx-6D)

- 1.1. Ось 1 (Интенция): [...]
- 1.2. Ось 2 (Действие): [...]
- 1.3. Ось 3 (Объект): [...]
- 1.4. Ось 4 (Масштаб): [...]
- 1.5. Ось 5 (Риск): [...]
- 1.6. Ось 6 (Модус): [...] или Н/П

БЛОК 2: СОДЕРЖАНИЕ АНАЛИЗА

- 2.1. Наблюдение / Вывод: [...]
- 2.2. Обоснование / Аргументация: [...]
- 2.3. Теоретическая линза: [...] или Н/П

БЛОК 3: СИСТЕМНАЯ ИНТЕГРАЦИЯ

- 3.1. Первичный код / Понятие: [...]
- 3.2. Связь с гипотезами (ID): [Подтверждает H-__ / Опровергает H-__ / Модифицирует H-__]

БЛОК 4: МЕТА-АНАЛИЗ (обязателен)

- 4.1. Аналитическая рефлексия: [сомнения, альтернативы, аффективный отклик, методологические заметки]
```

## Дисциплина теоретических линз (Manual: 1.1.3; App.6)

### Deferred selection (отложенный выбор)

- В **Фазе 1** по умолчанию держать **Блок 2.3 = Н/П** и оставаться на Axis5=1–2 (дескрипция/логико-структурный вывод).
- Линзу фиксировать в аннотации только тогда, когда вы **делаете модельно-нагруженный вывод**, который должен быть проверяемым (обычно Фаза 2).

### Правило риска: линза ⇒ Axis5 ≥ 3

Если в аннотации заполнен **Блок 2.3 “Теоретическая линза”**, то:

- **Ось 5 (Риск) должна быть ≥ 3** (абдуктивный уровень или выше),
- в **Блоке 4** обязательно указать хотя бы 1 сильную альтернативу (counter-lens / иной конструкт), которую нужно проверить.

### Плюрализм (pluralism)

Для ключевого феномена/паттерна:

- создать минимум **2 аннотации** с **разными линзами** (например, Primary и Counter из Lens Selection Framework),
- явно указать, *какие разные предсказания* дают линзы (что должно/не должно встретиться в данных).

### Адекватность линзы (lens adequacy check)

Линза считается “адекватно применённой” только если она:

- генерирует **фальсифицируемые** H1/H0 (см. Skill `ddx-hypothesis-lifecycle`, Manual: 1.1.4),
- позволяет найти **дискриминанты** (Manual: 1.3.3),
- не превращается в “объяснение задним числом” без проверяемых предсказаний.

## Алгоритм кодирования (5 шагов, обязательный порядок)

### Шаг 1: Интенция (Ось 1) -- "Зачем я выделяю этот фрагмент?"

| Код | Интенция | Вопрос-подсказка |
|-----|----------|-----------------|
| 1.1 | Эксплорация | Ищу новое, неизвестное? |
| 1.2 | Генерация | Создаю гипотезу / альтернативу? |
| 2.1 | Валидация | Проверяю, подтверждаю/опровергаю? |
| 2.2 | Оценка | Сравниваю, ранжирую? |
| 3.1 | Оптимизация | Уточняю, углубляю существующее? |
| 3.2 | Интеграция | Синтезирую разрозненное в целое? |
| 4.1 | Калибровка | Анализирую сам процесс / себя? |

### Шаг 2: Объект + Масштаб (Оси 3, 4) -- "На что смотрю и на каком уровне?"

**Объект (Ось 3):** A.1 Единичные данные / A.2 Структурированные данные / B.1 Гипотеза / B.2 Каузальная модель / C.1 Логика вывода / C.2 Субъект диагностики

**Масштаб (Ось 4):** 1 Суб-элементный / 2 Элементный / 3 Комплексный / 4 Системный / 5 Мета-системный

### Шаг 3: Действие (Ось 2) -- "Какую операцию применяю?"

1.0 Рефрейминг (смена перспективы) / 2.0 Деконструкция (расчленение) / 3.0 Трансформация (пересборка)

### Шаг 4: Риск (Ось 5) -- "Насколько далеко от прямого наблюдения?"

1 Дескриптивный (факт) / 2 Логико-структурный (вывод) / 3 Абдуктивный (гипотеза) / 4 Аналоговый (перенос) / 5 Трансгрессивный (контринтуитивный)

### Шаг 5: Модус (Ось 6) -- "Могу ли обосновать динамическую характеристику?"

Если да -- выбрать из: активность (норма/гипо/гипер), качество (дис-/пара-/цикл), организация (ригидность/слитность/фрагментация), динамика (эскалация/рекурсия/эмерджентность). Если нет данных -- `Н/П`.

## 8D-расширение (когда активирован реляционно-клинический режим)

Добавить к Блоку 1 после Оси 6:

```
- 1.7. Ось 7 (Ситуационный контекст): 7.1: [микро] / 7.2: [мезо] / 7.3: [макро]
- 1.8. Ось 8 (Реляционный контекст): 8.1: [интра] / 8.2: [интер] / 8.3: [системный] / 8.4: [символический]
```

Активировать 8D если: текст реляционный / нужна дифференциация диспозиция vs ситуация. Подробнее -- Skill `ddx-contextual-8d`.

## Быстрое кодирование (Basic-протокол)

Когда времени мало (Протокол A), обязательный минимум:

- **Ось 3 (Объект):** что анализируем
- **Ось 5 (Риск):** насколько надежно
- **Вывод (Блок 2.1):** что получили
- **Рефлексия (Блок 4):** хотя бы 1 предложение

Остальные оси -- по возможности.

---

## Масштабирование (большие корпуса): index-first + chunking

Если аннотаций много, не сохраняйте “всё в одном файле”.

- В кейсе вести **индекс**: `Artifacts/DDx/<case_id>/02_annotations/index.md` (anomalies-first + top representatives).
- Аннотации хранить чанками: `A-0001_A-0100.md`, `A-0101_A-0200.md` и т.д.

Эталон структуры: `Artifacts/DDx/_TEMPLATES/`.

### Пример 1: Фаза 1, дескриптивный

> Фрагмент: "Я опять не справился, как и всегда"

- 1.1 Ось 1: 1.1.1 Сканирование
- 1.3 Ось 3: А.1.1 Вербальные данные
- 1.4 Ось 4: 2 Элементный
- 1.5 Ось 5: 1 Дескриптивный
- 2.1 Наблюдение: Самообесценивание + генерализация неудачи ("как и всегда")
- 3.1 Код: #самообесценивание #генерализация

### Пример 2: Фаза 2, интерпретативный

> Фрагмент: "Я опять не справился, как и всегда"

- 1.1 Ось 1: 1.2.1 Идеация
- 1.3 Ось 3: B.1 Диагностическая гипотеза
- 1.5 Ось 5: 3 Абдуктивный
- 2.1 Наблюдение: Паттерн выученной беспомощности (Seligman)
- 2.3 Линза: Когнитивно-поведенческая → Теория выученной беспомощности → Конструкт: атрибутивный стиль
- 3.2 Связь: Подтверждает H-03

Связанные правила: `ddx-annotation.mdc`, `ddx-axes-formula.mdc`, `ddx-8d-context.mdc`, `ddx-phases.mdc`.
