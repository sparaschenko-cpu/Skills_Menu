# Полный DDx-анализ (Run Analysis)

Этот навык -- оркестратор. Он проводит через все фазы анализа, ссылаясь на специализированные навыки для деталей.

**Делегирование субагентам:** для изоляции контекста и параллельной работы делегируй фазовые задачи субагентам из `.cursor/agents/`. Субагенты работают в собственных контекстных окнах. См. таблицу в `ddx-workspace-context.mdc`. Ключевой порядок: `/ddx-guardian` (GP-0) -> `/ddx-explorer` (Phase 1) -> `/ddx-guardian` (GP-1) -> `/ddx-analyst` (Phase 2) -> `/ddx-guardian` (GP-2..GP-3) -> `/ddx-synthesizer` (Phase 3) -> `/ddx-guardian` (GP-5..GP-6).

## Шаг 0: Подготовка (Phase 0)

### 0.1 Выбор протокола

| Протокол | Когда | Объём |
|----------|-------|-------|
| **A (Basic)** | Экспресс, жёсткий дедлайн, обучение | Минимальные поля + 1 цикл H1/H0 |
| **B (Standard)** | Основной режим, глубокое заключение | Полные 3 фазы + все оси + СМГ |
| **C (Max)** | Судебная экспертиза, высокие ставки | Standard + все режимы + внешняя супервизия |

Детали каждого протокола -- [protocol-checklists.md](protocol-checklists.md).

### 0.2 Формулирование диагностического вопроса

Конкретно: не "проанализировать текст", а "выявить механизм, приводящий к [X] в [контексте Y]".

### 0.3 Определение границ

- **Ось 3 (Объект):** что именно является фокусом (личность / система / процесс)?
- **Ось 4 (Масштаб):** на каком уровне работаем (микро / мезо / макро)?

### 0.4 Политика риска (Ось 5)

Заранее решить: какой максимальный уровень риска допустим для центральных выводов?

### 0.5 Safety-проверка

Выполнить сокращённый Zero Point (полный -- Skill `ddx-safety-audit`):

- [ ] Аффективное состояние: в норме?
- [ ] Теоретические преференции: осознаны?
- [ ] Первичный резонанс: зафиксирован?

### 0.6 Решение о 8D

Нужен ли 8D-режим? Если текст реляционный или требуется диспозиция vs ситуация -- активировать (Skill `ddx-contextual-8d`).

## Шаг 1: Эксплорация (Phase 1)

**Цель:** "карта данных" -- структурированное описание всего материала.

### Процедура

1. **Полное прочтение** материала без анализа.
2. **Открытое кодирование:** для каждого значимого фрагмента создать аннотацию (Skill `ddx-annotate-and-code`). Риск строго 1-2.
3. **Кластеризация:** сгруппировать коды по тегам (#) в тематические кластеры.
4. **Выявление аномалий:** противоречия, лакуны, "выбросы".
5. **Формирование первичных понятий:** для каждого крупного кластера -> обобщающее понятие.
6. **Построение Карты данных** (шаблон в [protocol-checklists.md](protocol-checklists.md)).
7. **Выбор первичной оптики (Standard/Max):** применить Lens Selection Framework (Прил.6) к результатам скрининга:
   - выбрать **Primary lens** + **Counter-lens** (и при необходимости 3-ю комплементарную),
   - зафиксировать *Paradigm → Theory → Construct → Model* и критерий интеграции,
   - зафиксировать собственные преференции и обязательство проверить минимум 1 сильную гипотезу через counter-lens.
   Детали: `ddx-lens-selection-framework.mdc`.

### Критерии перехода к Фазе 2

- [ ] Весь материал проаннотирован
- [ ] Коды кластеризованы
- [ ] Аномалии зафиксированы
- [ ] Первичные понятия сформулированы
- [ ] Карта данных построена
- [ ] Выбраны 2-3 комплементарные теоретические линзы с обоснованием (Прил.6) (для Standard/Max)

## Шаг 2: Экспликация (Phase 2)

**Цель:** система проверенных гипотез.

### Процедура

1. **Генерация гипотез** из карты данных: запустить Skill `ddx-hypothesis-lifecycle` (раздел "Генерация"). Использовать все 3 источника + Ось 2.
2. **Формализация** каждой гипотезы в H-card. Обязательно H1/H0 пары.
3. **Итеративные циклы "генерация <-> фальсификация":** Fast Falsification (Skill `ddx-hypothesis-lifecycle`, раздел 4) для каждой значимой пары.
4. **Байесовское обновление** после каждого значимого свидетельства.
5. **Safety-проверка:** для каждой H-Pathology -> H-Adaptive (Skill `ddx-safety-audit`).
6. **При пуле > 7-10 гипотез:** запустить Воронку Валидации (Skill `ddx-hypothesis-lifecycle`, раздел 6).

### Переключающие протоколы (из `ddx-phases.mdc`)

| Ситуация | Действие |
|----------|---------|
| После дивергенции | -> обязательная конвергенция (валидация) |
| Обнаружена аномалия | -> зондирование (переключение с 1.1.1 на 1.1.2) |
| Гипотеза фальсифицирована | -> генерация альтернативы (1.2.2) |
| Высокий риск (4-5) | -> обязательная низкорисковая проверка |
| Аналитический тупик | -> мета-позиция (4.1 Калибровка) |
| Тупик не снимается | -> Mode 1 (Skill `ddx-activate-mode`) |

### Критерии перехода к Фазе 3

- [ ] Центральная H1 прошла fast falsification
- [ ] Основные конкуренты рассмотрены и аргументированно отклонены / интегрированы
- [ ] Байесовский статус стабилизировался (нет резких скачков)

## Шаг 3: Интеграция (Phase 3)

**Цель:** целостная динамическая модель с прогностической ценностью.

### Процедура

1. **Структурная карта** гипотез: иерархия + сеть (`ddx-smh-advanced.mdc`).
2. **Динамическая квалификация** через Ось 6 (`ddx-integration-conclusion.mdc`).
3. **Углублённые сценарии** (если Standard/Max):
   - 2.2.1 Системная динамика + петли
   - 2.2.2 Онтогенез
   - 2.2.3 Кросс-масштабный анализ
   - 2.2.4 Коморбидность
   - Детали: `ddx-deep-scenarios.mdc`
4. **Серийные данные** (если есть):
   - 2.3.1 Трекинг изменений
   - 2.3.2 Прогностическое моделирование
5. **Проверка на pathology-only:** если модель однобокая -> Mode 2 (Skill `ddx-activate-mode`).

## Шаг 4: Финализация

### 4.1 Pre-conclusion safety

Запустить Skill `ddx-safety-audit` (раздел "Pre-conclusion: Internal Red Teaming").

Для Max-протокола: Mode 3 (Skill `ddx-activate-mode`) -> Паспорт надежности.

### 4.2 Написание заключения

Запустить Skill `ddx-write-conclusion`. Следовать шаблону 3.3.3 и чек-листу качества.

### 4.3 Архивация

Сохранить: все аннотации, СМГ-реестр, H-cards, Байесовские логи, Карту данных, выбранные линзы (Primary/Counter + обоснование), Матрицу доказательств (если строилась), структурные карты, заключение.
