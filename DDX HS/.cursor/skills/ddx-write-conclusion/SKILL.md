# Диагностическое заключение (Write Conclusion)

## Шаблон заключения (3.3.3, copy-paste)

```
ИТОГОВОЕ ДИАГНОСТИЧЕСКОЕ ЗАКЛЮЧЕНИЕ

Дата: [...]  ID Случая: [...]  Аналитик(и): [...]
Исходные материалы: [...]
Диагностический вопрос / Цель анализа: [...]

РАЗДЕЛ 1: РЕЗЮМЕ КЛЮЧЕВЫХ ВЫВОДОВ
[Сжатое изложение центральной модели, динамик, прогноза. Каждый вывод квалифицирован по Оси 5.]

РАЗДЕЛ 2: СТРУКТУРНО-ФУНКЦИОНАЛЬНАЯ МОДЕЛЬ (Морфогенез)
- 2.1. Центральная диагностическая модель: [...]
- 2.2. Ключевые подсистемы и их организация: [сетевая диаграмма]
- 2.3. Сопутствующие (коморбидные) феномены: [...]

РАЗДЕЛ 3: ДИНАМИЧЕСКАЯ МОДЕЛЬ
- 3.1. Ключевые модусы функционирования (Ось 6): [...]
- 3.2. Основные петли обратной связи и системная динамика: [диаграмма]

РАЗДЕЛ 4: ИСТОРИКО-ГЕНЕТИЧЕСКАЯ МОДЕЛЬ (Онтогенез и Этиология)
- 4.1. Реконструкция онтогенеза: [нарратив + таймлайн]
- 4.2. Этиологическая гипотеза: [...]

РАЗДЕЛ 5: ПРОГНОСТИЧЕСКАЯ ОЦЕНКА
- 5.1. Инерционный сценарий: [...]
- 5.2. Факторы риска и уязвимости: [...]
- 5.3. Факторы резильентности и ресурсы: [если Mode 2 активирован]
- 5.4. Потенциал к изменению: [если Mode 2 активирован]

РАЗДЕЛ 6: МЕТОДОЛОГИЧЕСКОЕ ПРИЛОЖЕНИЕ
- 6.1. Ключевые рассмотренные альтернативы: [отвергнутые гипотезы + Axis5 4-5 выводы]
- 6.2. Ограничения анализа: [+ Паспорт надежности, если Mode 3 активирован]
- 6.3. Аналитическая рефлексия: [...]

ПРИЛОЖЕНИЕ: СПИСОК ДОКАЗАТЕЛЬСТВ
[Таблица: Утверждение -> ID Аннотаций]
```

## Пошаговый workflow написания

### Шаг 1: Сбор материала

Из СМГ-реестра (Skill `ddx-hypothesis-lifecycle`) собрать все гипотезы со статусом "Подтверждённая" и "Рабочая". Для каждой -- проверить наличие H-card с заполненным Блоком 3 (свидетельства).

### Шаг 2: Структурная карта (-> Раздел 2)

Построить иерархическую модель (причинные цепи, fishbone) + сетевую модель (узлы + связи +/-). Процедуры построения -- `ddx-smh-advanced.mdc`, раздел "Процедуры построения моделей (1.3.2)".

**Дисциплина связей (Manual: 1.3.3):** для **каждой** стрелки/связи/петли, которую вы утверждаете в структурной или динамической модели, требуется:

- 1–3 опорных **A-ID** (где оба элемента проявлены в одном контексте),
- краткое основание связи по корреляционному протоколу: совместная встречаемость + (по возможности) последовательность/условность/интенсивность,
- пометка Axis5 для уровня уверенности в связи.

Правило: **“стрелка без корреляционного основания” недопустима**.

### Шаг 3: Динамическая квалификация (-> Раздел 3)

Для ключевых узлов и связей применить Ось 6: активность, качество/паттерн, организация, динамика. Каждый модус -- на основании наблюдаемых паттернов.

### Шаг 4: Онтогенез (-> Раздел 4)

Реконструировать историю развития: таймлайн + ключевые события + траектории. Если Сценарий 2.2.2 выполнен -- интегрировать его результат. Детали: `ddx-deep-scenarios.mdc`.

### Шаг 5: Прогнозы (-> Раздел 5)

Сформулировать 2-4 сценария будущего (инерционный, оптимистичный, пессимистичный, трансформационный). Квалифицировать по Оси 5. Если Сценарий 2.3.2 выполнен -- интегрировать.

### Шаг 6: Написание текста

Для **каждого** материального утверждения в Разделах 1-5:
- Дать ссылку на конкретные **annotation IDs** (A-___).
- Указать **Ось 5** (эпистемический статус).

### Шаг 7: Методологическое приложение (-> Раздел 6)

- 6.1: Описать все отвергнутые гипотезы с обоснованием. Все выводы Axis5 4-5, не прошедшие независимое подтверждение.
- 6.2: Ограничения (данные, методы, контекст). Если Mode 3 был активирован -- включить **Паспорт надежности** (шаблон в `ddx-resource-metacritique.mdc`).
- 6.3: Аналитическая рефлексия (главные сомнения, влияние личности аналитика).

---

## Чек-лист качества заключения

Перед финализацией проверить:

- [ ] Все утверждения в Разделах 1-5 имеют ссылку на annotation IDs?
- [ ] Все утверждения имеют маркировку Оси 5?
- [ ] Рассмотрены и описаны альтернативные гипотезы (Раздел 6.1)?
- [ ] Выводы Axis5 4-5 размещены в 6.1, а не в основных разделах (если не подтверждены Риском 1-3)?
- [ ] Есть прогностический раздел (5)?
- [ ] Есть аналитическая рефлексия (6.3)?
- [ ] Модель НЕ pathology-only? (Если да -- триггер Mode 2, Skill `ddx-activate-mode`)
- [ ] Если Mode 2 активирован: заполнены 5.3 и 5.4?
- [ ] Если Mode 3 активирован: Паспорт надежности включен в 6.2?
- [ ] Pre-conclusion Red Teaming пройден (Skill `ddx-safety-audit`)?

---

## Интеграция с Ресурсной картой (Mode 2)

Если Mode 2 был активирован (Skill `ddx-activate-mode`), его результат (SWOT-матрица + Ресурсная карта) включается в:
- **Раздел 5.3** "Факторы резильентности и ресурсы"
- **Раздел 5.4** "Потенциал к изменению"

Если Mode 2 НЕ активирован -- проверить, не стала ли модель pathology-only. Если да: запустить Mode 2 перед финализацией.

## Интеграция с Паспортом надежности (Mode 3)

Если Mode 3 был активирован, включить Паспорт надежности (5 полей: ядро, уязвимости, чувствительность, границы, оценка) в **Раздел 6.2**.

Связанные правила: `ddx-integration-conclusion.mdc`, `ddx-smh-advanced.mdc`, `ddx-deep-scenarios.mdc`, `ddx-resource-metacritique.mdc`, `ddx-safety.mdc`.
